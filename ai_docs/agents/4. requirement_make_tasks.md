# Requirement Make Tasks Agent

Transform a validated requirement and its solution design into actionable development tasks.

## Purpose

Break down a solution design into granular, implementable tasks that developers can execute following the project's development workflow.

## Prerequisites

Before creating tasks:
1. Requirement must be validated (Agent 2)
2. **MANDATORY:** Solution design document must exist (Agent 3)
3. Read `/ai_docs/architecture/development-workflow.md` - Implementation phases

## Input

- **MANDATORY:** Path to the solution design document (`design.md`)
- Path to the validated requirement

## Instructions

### Step 0: Verify Design Document Exists

**MANDATORY - DO NOT SKIP**

Before creating any tasks, verify that the design document exists:
- Location: `/docs/working_docs/[type]/[name]/design.md`
- If design.md does NOT exist → **STOP** and ask user to run Agent 3 first
- Tasks cannot be created without a solution design

### Step 1: Read Documentation

1. **MANDATORY:** Solution design document (`design.md`) - **READ THIS FIRST**
2. `/ai_docs/architecture/development-workflow.md` - Implementation phases
3. `/ai_docs/architecture/architecture.md` - DDD structure
4. Original requirement document

### Step 2: Extract Implementation Items

From the solution design, extract:
- Domain entities, value objects, enums
- Commands and queries
- Repository interfaces and implementations
- HTTP layer components (actions, requests, resources)
- Migrations
- Tests
- Collateral changes

### Step 3: Apply Development Phases

Tasks MUST follow the development workflow phases:

```
PHASE 1: Domain Layer
  ├── Entities
  ├── Value Objects
  └── Enums

PHASE 2: Infrastructure Layer
  ├── Repository Interface (Domain)
  ├── Repository Implementation
  ├── Mapper/Hydrator
  ├── ORM Model
  └── Migrations

PHASE 3: Application + HTTP Layer
  ├── Commands + Handlers
  ├── Queries + Handlers
  ├── Request classes
  ├── Action classes
  ├── Resources
  ├── Controller
  └── Routes
```

### Step 4: Create Tasks

For each implementation item, create a task with:
- Clear title
- Description
- Acceptance criteria
- Dependencies on other tasks
- Estimated complexity (S/M/L)
- Phase reference

### Step 5: Order Tasks

Tasks must be ordered by:
1. Phase (Domain → Infrastructure → Application)
2. Dependencies (dependent tasks after their dependencies)
3. Priority (critical path first)

## Task Complexity Guide

| Complexity | Description | Typical Time |
|------------|-------------|--------------|
| **S** | Simple, straightforward | < 1 hour |
| **M** | Medium, some complexity | 1-4 hours |
| **L** | Large, significant work | 4-8 hours |
| **XL** | Very large, consider splitting | > 8 hours |

## Tips

- Keep tasks small and focused
- One task = one responsibility
- Include file paths for clarity
- Make acceptance criteria testable
- Don't forget collateral changes
- Always include testing tasks
