# Requirement Slicing Agent

Divide a large, validated requirement into smaller, independently deliverable slices.

## Purpose

When a requirement is too large to implement in a single iteration, this agent helps break it down into smaller pieces (slices/features) that can be delivered incrementally while maintaining business value.

## When to Use

**OPTIONAL AGENT** - Use only when:
- Agent 2 (Validator) flagged the requirement as "too large" or "slicing needed"
- The requirement has 4+ entities, 10+ use cases, or complex state machines
- You want to deliver value incrementally rather than in one big release
- Different parts of the requirement have different priorities

**Skip when:**
- Requirement is already small enough (1-3 entities, 3-7 use cases)
- Hotfixes or Cases (these don't get sliced)
- Features (already a slice of an epic)

## Input

- Path to the **validated** requirement document (should have passed Agent 2)
- Optionally: preferred slicing strategy

$ARGUMENTS

## Prerequisites

Before slicing:
1. Requirement must be validated (Agent 2 status: "Valid" or "Approved")
2. All critical questions from validation should be resolved
3. Business alignment and KPIs should be clear

## Instructions

### Phase 1: Read Documentation

1. `/ai_docs/development_process/requirement_analysis_guide.md` - Section 8: Slicing
2. Original requirement document
3. Validation report (if exists)

### Phase 2: Analyze Requirement Size

Assess the requirement complexity:

| Indicator | Small | Medium | Large | XL |
|-----------|-------|--------|-------|-----|
| Entities | 1-2 | 3-4 | 5-7 | 8+ |
| Use Cases | 1-5 | 6-10 | 11-20 | 20+ |
| States per entity | 2-3 | 4-5 | 6-8 | 9+ |
| Integrations | 0-1 | 2-3 | 4-5 | 6+ |
| User roles affected | 1 | 2-3 | 4-5 | 6+ |

**Recommendation:**
- Small/Medium: No slicing needed
- Large: Recommend 2-3 slices
- XL: Recommend 3-5 slices

### Phase 3: Identify Slicing Strategy

Choose the most appropriate strategy based on the requirement nature:

#### Strategy 1: By CRUD Progression
Best for: Entity-centric requirements

```
Slice 1: Create + Read (MVP)
Slice 2: Update + Delete
Slice 3: Bulk operations + Advanced queries
```

#### Strategy 2: By User Role
Best for: Multi-stakeholder requirements

```
Slice 1: End user features
Slice 2: Admin features
Slice 3: Super admin / System features
```

#### Strategy 3: By Workflow Stage
Best for: Process-oriented requirements

```
Slice 1: Creation/Initiation
Slice 2: Processing/Execution
Slice 3: Completion/Reporting
```

#### Strategy 4: By Priority (MoSCoW)
Best for: Time-constrained requirements

```
Slice 1: Must Have (MVP)
Slice 2: Should Have (Enhanced)
Slice 3: Could Have (Full)
Slice 4: Won't Have (Future)
```

#### Strategy 5: By Risk
Best for: Uncertain or exploratory requirements

```
Slice 1: Core functionality (low risk)
Slice 2: Edge cases + error handling
Slice 3: Integrations + advanced features
```

#### Strategy 6: By Entity
Best for: Requirements with multiple independent entities

```
Slice 1: Primary entity complete
Slice 2: Secondary entity complete
Slice 3: Entity relationships + cross-entity features
```

### Phase 4: Define Slices

For each slice:

1. **Identify scope** - Which use cases belong to this slice?
2. **Ensure independence** - Can this slice work without the others?
3. **Define MVP per slice** - What's the minimum for this slice to deliver value?
4. **Map dependencies** - What from other slices is needed?
5. **Identify shared components** - What can be reused across slices?

### Phase 5: Validate Slices

For each slice, verify:

| Check | Question |
|-------|----------|
| **Independence** | Can this slice be deployed without others? |
| **Value** | Does this slice deliver business value on its own? |
| **Completeness** | Is the slice feature-complete for its scope? |
| **Testability** | Can this slice be tested independently? |
| **No orphan states** | Are all states in this slice reachable and have exits? |
| **No broken flows** | Do all user journeys in this slice have proper endings? |

### Phase 6: Handle Dependencies

Document inter-slice dependencies:

| Dependency Type | Example | How to Handle |
|-----------------|---------|---------------|
| **Data dependency** | Slice 2 needs entity from Slice 1 | Implement Slice 1 first |
| **Schema dependency** | Slice 2 adds fields to Slice 1's table | Design schema for both upfront |
| **State dependency** | Slice 2 adds new status | Reserve status values in Slice 1 |
| **UI dependency** | Slice 2 button goes on Slice 1's screen | Design UI layout for extensibility |
| **API dependency** | Slice 2 needs Slice 1's endpoint | Implement endpoint in Slice 1 |

### Phase 7: Generate Slice Documents

Create individual requirement documents for each slice.

## Important Principles

### 1. Each Slice Must Deliver Value
A slice is NOT just "part of the work". It must:
- Be deployable to production
- Provide business value to users
- Be testable end-to-end

### 2. Design for the Whole, Build in Parts
When designing Slice 1:
- Consider the schema needs of all slices
- Reserve enum values for future states
- Design extensible APIs
- Document what future slices will need

### 3. Avoid "Slice Debt"
Common mistakes:
- Slice 1 with Create, Slice 2 with Read (incomplete!)
- Slice without proper error handling
- Slice that requires another slice to be useful

### 4. Maintain Traceability
Every slice must link back to:
- Original requirement
- Slicing plan
- Related slices

---

## Philosophy: Analysis Informs, Never Blocks

**THE USER ALWAYS DECIDES.**

- If user disagrees with slicing strategy, adapt
- If user wants fewer/more slices, adjust
- If user wants to proceed without slicing, document concerns and proceed
- Slicing is a **tool, not a gate**

**NEVER say:** "This must be sliced into exactly 3 parts"
**ALWAYS say:** "I recommend 3 slices because [reason]. Proceed with this approach or adjust?"
